Metamarket Codebase Audit & Problems Report
==========================================

Context:
---------
Metamarket is a full-stack prediction market application. 
- Backend: Express.js (Node.js), MongoDB (Mongoose), Passport.js (Google Auth).
- Frontend: React (Vite), Tailwind CSS, React Query, React Router.
- Infrastructure: Configured for local development (Vite), standalone production (Docker/Node), and serverless (Vercel).

The server logic resides in the `mapi/` directory, while the Vercel entry point is in `api/`.

Identified & Addressed Problems:
--------------------------------

1. Broken Vite Dev Server Routing:
   - Problem: The Express app was mounted as a global middleware in Vite's `configureServer`. Express's default behavior is to send a 404 response if no route matches. This blocked Vite's own handlers from serving the frontend (SPA routes, HMR, assets).
   - Fix: Added a conditional check in `vite.config.ts` to only pass requests to the Express app if they start with `/mapi` or `/uploads`.

2. Redundant & Loop-prone Proxy:
   - Problem: `vite.config.ts` contained a proxy rule for `/mapi` targeting `localhost:8080` (the same port Vite was running on). This is redundant when the server is integrated and can cause request loops.
   - Fix: Removed the proxy configuration.

3. Unstructured API Routing:
   - Problem: API routes were defined directly on the top-level Express app with hardcoded `/mapi` prefixes scattered throughout `mapi/server/index.ts`.
   - Fix: Unified all API endpoints into a dedicated `apiRouter` and mounted it once at `/mapi`. This improves maintainability and ensures consistency.

4. Performance Inefficiency in Production:
   - Problem: In `node-build.ts`, the authentication middleware (which performs a MongoDB lookup) was running before static file serving. This meant every request for an image or script triggered a database query.
   - Fix: Reorganized the middleware stack and unified routes to ensure static assets are prioritized or handled correctly.

5. Type Mismatches in Build Tooling:
   - Problem: Using Express as a middleware inside Vite (which uses Connect) caused TypeScript errors because their Request/Response types differ slightly, despite being runtime-compatible.
   - Fix: Applied type casting to 'any' in `vite.config.ts` to satisfy the TypeScript compiler without affecting runtime behavior.

Status:
-------
The routing logic has been refactored and the local development environment is now synchronized with the production and serverless configurations.
